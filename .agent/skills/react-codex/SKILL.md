# React Codex (Project Workflow)

这是本项目 React 任务的统一工作流。凡是涉及 React 组件、TSX/JSX、Hook、shadcn/ui、
页面交互、UI 重构、渲染逻辑调整，都应先加载本 skill，再按下面的流程推进。

目标不是“知道用哪些 skill”，而是把一个 React 任务从接收需求到最终交付走完整。

## 已包含的现有 Skills

- `design-md`：基于 Google Labs design.md 的机器可读设计契约，统一 token 与设计边界。
- `shadcn`：shadcn/ui 组件查找、docs、CLI、registry、preset、组合规则与样式规范。
- `frontend-design`：页面视觉与交互改版，避免产出模板化界面。
- `vercel-react-best-practices`：React 性能、状态设计、渲染优化、模块拆分最佳实践。

页面视觉规范不要在本 skill 中重复维护。

当任务命中以下表达时，页面设计规范优先直接遵循 `design-md`：

- `简约`
- `大气`
- `高级感`
- `苹果风`
- `Apple`
- `克制`
- `留白`

## 默认调用方式

优先一次性加载：

```bash
npx openskills read react-codex,design-md,shadcn,frontend-design,vercel-react-best-practices
```

如果当前任务比较简单，也可以先只加载本 skill，再按阶段补充：

```bash
npx openskills read react-codex
```

## 适用范围

命中以下任一情况，就按本流程执行：

- React 页面、组件、Hook、TSX/JSX 编写或修改
- shadcn/ui 组件接入、修复、替换、样式调整
- 页面布局改版、交互增强、视觉统一
- 列表、表单、弹窗、Tabs、Table、Card、Drawer 等典型 UI 工作
- 渲染性能、状态管理、组件拆分、结构重构

以下场景通常不必走完整流程：

- 纯配置修改
- 纯样式文件微调
- 类型声明文件修改
- 单纯的问答、review、定位 bug 但暂不改代码

## 完整开发流程

### 第 0 阶段：任务分类

先判断当前任务属于哪一类，再决定要不要补充其他 skill：

1. shadcn/ui 组件开发/修复

```bash
npx openskills read react-codex,shadcn
```

2. 页面改版、视觉统一、交互增强

```bash
npx openskills read react-codex,design-md,frontend-design
```

3. 性能优化、渲染异常、重构

```bash
npx openskills read react-codex,vercel-react-best-practices
```

4. 复杂任务（UI + shadcn + 性能）

```bash
npx openskills read react-codex,design-md,shadcn,frontend-design,vercel-react-best-practices
```

### 第 1 阶段：理解任务边界

开始改代码前，先明确下面这些点：

1. 改的是页面、业务组件、基础组件，还是 Hook。
2. 需求是改视觉、改交互、改数据流，还是修 bug。
3. 哪些文件必须改，哪些文件尽量不动。
4. 是否涉及现有组件复用、共享状态、路由跳转、接口联动。
5. 是否存在设计稿、参考页面、既有页面风格可对齐。

如果能通过本地代码推断清楚，就直接推进；不要为了小问题频繁停下来问用户。

### 第 2 阶段：建立实现上下文

动手前先看清上下文，不要直接凭记忆改。

最低要求：

1. 打开当前目标文件。
2. 看它引用的组件、Hook、样式来源。
3. 查找同模块下相似页面/相似组件。
4. 看数据来源、状态归属、事件流向。
5. 明确当前是受控组件、派生状态，还是副作用驱动。

建议优先关注：

- 是否已有相同 UI 模式可复用
- 是否已有同风格页面可对齐
- 是否已有抽象层，不要重复造轮子
- 若是页面布局任务，先拆分页面模块，并判断哪些区域可直接由 shadcn/ui 组件组合完成
- 先梳理页面所需组件列表，再检查项目 `components/**` 中是否已有对应封装

### 第 3 阶段：查组件 API / Docs，不靠记忆

凡是涉及组件库，先查官方 API / docs / demo，再写代码。

#### shadcn/ui 固定流程

先确认项目上下文、已安装组件和官方 docs，再决定是否新增组件：

```bash
npx shadcn@latest info --json
npx shadcn@latest docs <component>
npx shadcn@latest search @shadcn -q "<keyword>"
```

遇到这些情况必须查：

- 不确定组件是否已经安装
- 不确定应该用哪个 shadcn 组件来组合页面
- 不确定当前组件的组合方式、结构约束或推荐用法
- 需要从 registry 新增组件或 block
- 需要更新已有 shadcn 组件但又要保留本地改动

涉及新增 / 更新时，优先按下面顺序：

```bash
npx shadcn@latest add <component> --dry-run
npx shadcn@latest add <component> --diff <file>
npx shadcn@latest add <component>
```

项目内组件检查顺序：

1. 先列出页面需要的 shadcn/ui 相关组件清单
2. 检查项目 `components/**` 中是否已经存在对应封装
3. 若已存在，先判断能否直接复用
4. 若不能直接复用，判断是否适合在现有组件基础上扩展
5. 若项目内不存在且确有必要，再新增封装

额外约束：

- 先看 docs 和当前项目 `info`，不要凭印象直接写 shadcn 组件结构。
- 优先复用现有 shadcn 组件，不先手写一个外观相似的 `div`。
- 页面布局任务先完成“页面模块清单 + 组件清单 + 项目组件检查”，再进入具体编码。
- 不要手动从 GitHub 抄组件源码，优先使用 `shadcn` CLI。
- 若用户要求加 block / registry 组件，必须确认 registry，不要替用户猜。

### 第 4 阶段：设计实现方案

编码前先在脑子里定清楚方案，至少包含：

1. 状态放在哪一层。
2. 哪些值应该 `useState`，哪些应该 `useMemo`/render 推导。
3. 哪些事件应直接触发更新，哪些副作用应放进 `useEffect`。
4. 哪些逻辑应抽函数，哪些应抽小组件。
5. 是否需要保留旧结构，还是可以重构。
6. 若使用 shadcn/ui，是复用现有组件、补装组件，还是组合已有 primitives。
7. 若是页面布局，页头、筛选区、内容区、侧栏、表单区、反馈区分别可落到哪些 shadcn/ui 组件。
8. 组件清单里哪些项目内已存在、哪些可扩展、哪些必须新封装。
9. 若需求偏简约/高级感，是否已按 `design-md` 规范进一步减少模块、压缩文案、降低色彩噪音。

约束：

- 优先类型安全，避免隐式 any。
- 避免随意使用 `Number` 转换数据，防止在大整数、高精度金额、ID 等场景中造成精度丢失；确需转换时，应先明确数据范围与精度要求，并通过显式校验或专用解析逻辑处理。
- 优先最小状态模型，避免冗余状态。
- 避免把可计算值塞进 state。
- 避免 effect 驱动本可同步推导的 UI。
- 能局部改，不做无关大改。

### 第 5 阶段：编码实现

执行时遵循下面规则：

1. 先做结构正确，再做样式完善。
2. 优先改动当前模块，不轻易扩散到全局。
3. 能复用既有组件就复用。
4. 复杂 JSX 适度拆小，避免一个组件塞满业务逻辑。
5. 若使用 shadcn/ui，优先遵循其组合规则、语义 token 与现有组件结构，不手搓近似替代物。
6. 对页面布局，优先顺序始终是：复用项目已有组件 -> 扩展已有组件 -> 新增组件封装。
7. 样式优先遵循项目已有方案；若任务要求用 UnoCSS，就优先用 UnoCSS。
8. 只有 UnoCSS 或组件 API 表达不了时，才补充内联样式或局部样式文件。
9. 若页面目标是简约大气，优先按照 `design-md` skill 做减法，再考虑补充视觉元素。

### 第 6 阶段：性能与结构复查

改完后至少过一遍下面这些问题：

1. 有没有不必要的重渲染。
2. 有没有多余的 `useEffect`。
3. 有没有本可以 render 推导却放进 state 的值。
4. 有没有闭包竞态、定时器泄漏、监听器未清理。
5. 列表渲染的 key 是否稳定。
6. 交互过程中是否有滚动、定位、联动竞态。
7. 大组件是否应该拆分。

涉及性能或复杂联动时，优先参考 `vercel-react-best-practices` 的思路。

### 第 7 阶段：最小验证

每次 React 改动后，至少做这些验证：

1. 类型是否通过。
2. 核心交互是否闭环。
3. 空态、加载态、异常态是否还正常。
4. 桌面端和移动端是否都能正常展示。
5. 若涉及 shadcn/ui，是否存在错误组合、漏装组件、错误 import、硬编码颜色或违反组件结构规则。
6. 新增组件是否本可由项目已有组件复用或扩展得到，避免重复封装。

建议最少检查：

- 点击、输入、切换、滚动、弹窗开关
- 条件渲染分支
- 列表为空和有数据两种情况

### 第 8 阶段：输出结果

最终说明应包含：

1. 做了什么。
2. 为什么这么做。
3. 风险点或边界。
4. 做了哪些验证。
5. 若没法验证，要明确说出没验证什么。

## 针对 shadcn/ui 任务的专门流程

遇到 shadcn/ui 任务时，默认按这个顺序：

1. `npx shadcn@latest info --json` 确认项目上下文与已安装组件
2. `npx shadcn@latest docs <component>` 查 docs、示例与推荐组合
3. 梳理当前任务的组件清单，并检查项目 `components/**` 是否已有对应封装
4. 先判断可否直接复用，不能复用再判断是否适合扩展
5. 必要时 `npx shadcn@latest search` 查 registry 中是否已有现成组件或 block
6. 若需新增或更新，先 `--dry-run` / `--diff`，再决定是否真正 `add`
7. 编写或修改 TSX
8. 回头检查 import、组合结构、语义 token、响应式与无障碍是否正确

不要在不查 docs / info 的前提下直接“凭经验拼 shadcn”。

## 针对页面改版任务的专门流程

遇到 UI/UX 改版时：

1. 先找同项目中风格最接近的页面
2. 先分析页面结构，拆出 hero、导航、功能区、列表区、表单区、反馈区等模块
3. 基于页面模块梳理 shadcn/ui 组件清单，例如 `Card`、`Tabs`、`Table`、`Dialog`、`Sheet`、`Form`、`Alert`、`Badge`、`Skeleton`
4. 检查项目 `components/**` 中是否已封装相关组件
5. 若已封装，先判断能否直接复用；若不能直接复用，再判断是否可以扩展
6. 若项目内没有且确有必要，再新增组件封装
7. 再决定是复用其结构、卡片、tabs、hero、section 标题，还是局部变体
8. 统一页面的 hero、导航、卡片、列表节奏，不要每块各写一套
9. 只有 shadcn/ui 现有组件和项目内封装都无法合理表达时，再补自定义结构
10. 视觉改动不要破坏交互与滚动联动

如果目标是简约大气 / Apple 风，额外执行以下检查：

1. 首屏是否只有一个真正的视觉中心
2. 标题、正文、按钮数量是否还能继续减少
3. 是否违反了 `design-md` skill / `DESIGN.md` 中的配色、排版、留白和材质原则
4. 页面是否主要依靠排版、留白、材质和对齐，而不是靠装饰图层撑气质
5. 模块之间是否留有足够呼吸感，而不是每一屏都塞满内容

不要只改单个块的样式而忽略整页层级。

## 针对重构/优化任务的专门流程

优先检查：

- 状态是否过多
- effect 是否滥用
- 组件职责是否混杂
- 交互逻辑是否存在竞态
- 滚动联动、动画、异步请求是否有时序问题
- 是否可通过提取函数/局部组件提升可读性

## 强约束

1. React 任务不要跳过“先加载 react-codex”。
2. shadcn 任务不要跳过“先查 info / docs / 已安装组件”。
3. 页面布局任务先分析页面结构，并梳理可复用的 shadcn/ui 组件清单。
4. 页面布局任务先检查项目 `components/**` 里是否已有相关封装，再决定复用、扩展还是新增。
5. 若用户要简约大气 / Apple 风，必须优先读取并遵循 `design-md` skill，而不是在当前 skill 中临时发挥。
6. 不因为赶时间就跳过最小验证。
7. 不做与任务无关的大范围重构。
8. 如果已经发现自己偏离流程，应立即回到对应阶段重新执行。

## 交付标准

只有同时满足以下条件，才算完成一个 React 任务：

1. 需求已落地到代码。
2. 方案与项目现有结构兼容。
3. 关键交互未回归。
4. 说明里讲清楚“做了什么、为什么、怎么验证的”。
